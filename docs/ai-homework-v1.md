# AI 作业 App V1 需求与架构草案

## 1. 问题定义

当前一年级作业管理常见问题：

- 作业来源分散：微信群、纸质作业、口头通知、不同学科老师分别布置。
- 家长执行成本高：需要自己整理今天做什么、做到什么程度、哪里没掌握。
- 学生薄弱点不清晰：知道“错了”，但不知道是“拼音声母混淆”“20 以内进位加法不熟”“看图写话表达弱”。
- 老师反馈难聚合：单次作业能看见结果，但很难持续追踪某个学生一段时间内的能力变化。

这个产品的核心不是“再做一个作业列表”，而是：

**把分散作业结构化，并把作业结果映射成可追踪的能力标签，帮助家长和老师发现学生的真实薄弱点。**

## 2. 目标用户

V1 先聚焦单一场景，避免目标过散：

- 主要用户：一年级学生家长
- 次要用户：班主任/学科老师
- 管理对象：单个班级内的学生，或家庭内 1-2 个孩子

## 3. 核心使用场景

### 场景 A：家长汇总当天作业

家长把微信群文字、截图、语音转文字后的内容导入系统，系统自动识别：

- 学科
- 作业类型
- 截止时间
- 需要准备的材料
- 是否需要提交图片/视频/纸质内容

输出“今日作业清单”。

### 场景 B：家长记录完成情况

家长可以对每项作业记录：

- 已完成 / 未完成
- 用时
- 错题数
- 是否需要家长辅导
- 孩子卡住的具体点

### 场景 C：系统识别薄弱点

系统根据作业表现，把问题归因到能力标签，例如：

- 语文：拼音、识字、组词、造句、阅读理解、书写规范
- 数学：数感、口算、进位退位、应用题理解、单位换算
- 英语：字母认读、单词拼读、听力反应

一段时间后输出：

- 高频出错能力点
- 最近改善的能力点
- 需要强化练习的优先级

### 场景 D：老师快速查看班级趋势

老师可以看到：

- 班级共性薄弱点
- 某学生近期持续薄弱点
- 哪类作业完成率低

## 4. V1 范围

### Must Have

- 学生、家长、老师三类账号
- 作业录入：文本、图片 OCR
- AI 解析作业内容并结构化
- 今日/本周作业列表
- 作业完成状态记录
- 错题/困难点记录
- 能力标签体系
- 学生薄弱点分析页
- 老师查看学生/班级简报

### Should Have

- 微信截图批量导入
- 自动生成家长提醒
- 简单练习建议

### Could Have

- 个性化练习题自动生成
- 语音输入
- 与校方系统对接

### Won't Have in V1

- 复杂排课
- 多学校多年级通用平台化能力
- 直播/视频课程
- 教材全量知识图谱

## 5. V1 成功标准

如果 V1 有价值，至少应满足：

- 家长录入一份作业，1 分钟内能生成结构化清单
- 80% 以上常见作业内容可被正确分类
- 系统能对学生近 2 周表现输出前 3 个薄弱点
- 家长能明显减少“今天到底要做什么”的整理成本

## 6. 关键需求细化

### 6.1 作业采集

输入方式：

- 粘贴微信群文本
- 上传截图
- 手动新增

系统处理：

- OCR 提取文字
- LLM 解析作业项
- 标准化字段输出

标准字段建议：

- date
- subject
- title
- task_type
- description
- required_submission
- due_time
- estimated_duration
- source_channel

### 6.2 完成记录

每个作业项支持：

- completion_status
- spent_minutes
- error_count
- parent_note
- teacher_feedback
- difficulty_level

### 6.3 薄弱点识别

分析逻辑建议分两层：

1. 规则层
- 根据作业类型、错题描述、老师反馈映射到能力标签

2. AI 层
- 对家长自由文本、老师点评做语义归类
- 例如“加减法总是看错符号”归到“运算符识别/审题注意力”

### 6.4 结果展示

家长端：

- 今日作业
- 本周完成情况
- 学生薄弱点 Top N
- 建议复习方向

老师端：

- 班级作业完成率
- 学生风险名单
- 班级共性薄弱点

## 7. 非功能约束

- 数据安全：涉及未成年人数据，默认最小化采集
- 可解释性：薄弱点结论要能回溯到具体作业和记录
- 成本可控：优先使用“小模型/规则 + 大模型补充”的混合策略
- 移动优先：家长大概率在手机上使用
- 响应速度：查看列表和分析页应在 2 秒内完成

## 8. 推荐架构

V1 建议采用 **单体应用 + AI 服务模块**，不要一开始做微服务。

### 前端

- 家长端/老师端：Next.js
- UI：移动优先响应式页面

### 后端

- API：Next.js Route Handlers 或 NestJS
- 鉴权：手机号验证码 / 微信登录（二选一，V1 推荐先手机号）
- 异步任务：队列处理 OCR 和 AI 分析

### 数据层

- PostgreSQL：主业务数据
- Redis：任务队列、缓存、限流
- 对象存储：作业截图、提交图片

### AI 能力层

- OCR：调用现成 OCR 服务
- 结构化解析：LLM 将原始作业内容转成 JSON
- 薄弱点归因：规则引擎优先，LLM 做补充分类

## 9. 核心数据模型

建议的核心实体：

- User
- Student
- ParentStudentRelation
- TeacherStudentRelation
- Homework
- HomeworkItem
- HomeworkSubmission
- PerformanceRecord
- SkillTag
- WeaknessInsight

关键关系：

- 一个学生有多条作业
- 一条作业可拆成多个作业项
- 一个作业项可关联多个能力标签
- 一个学生在时间窗口内生成多条薄弱点洞察

## 10. 分阶段实施建议

### Phase 1: 能用

- 完成账号体系
- 完成作业录入与列表
- 完成 OCR + AI 结构化

### Phase 2: 有价值

- 完成作业完成记录
- 建立能力标签体系
- 输出学生薄弱点分析

### Phase 3: 有差异化

- 自动生成练习建议
- 老师班级视图
- 趋势分析与预警

## 11. 当前最合理的 MVP

如果你现在要最快开始，我建议 MVP 只做 3 件事：

1. 家长上传作业截图或粘贴文本
2. 系统自动生成今日作业清单
3. 家长勾选完成情况，系统输出学生薄弱点 Top 3

这是最小闭环，因为它同时验证了三个关键假设：

- 用户愿不愿意录入作业
- AI 能不能稳定结构化作业
- 薄弱点分析是否真的有感知价值

## 12. 目前最需要尽快确认的 5 个问题

这些问题会直接影响后续产品和技术路线：

1. 主要付费方是谁，是家长、学校还是老师？
2. 首发场景是家庭自用，还是班级/机构统一使用？
3. 作业主要来源是微信群截图，还是老师后台直接录入？
4. 你要先做中国大陆场景，还是更通用的国际化场景？
5. 你希望 AI 的价值重点放在“汇总作业”还是“分析薄弱点”？
